Scheduler Investigation

Month Planner · Template Team Added After Planning

Investigation target: a planned month where the template was later modified to add a required team, Song Leaders. Existing Counters and Ushers scheduling works, but Song Leaders scheduling says “No planned services in this month…”.

Reviewed code paths in getMonthPlan, planMonthInternal, saveTemplate, and MonthSchedulePanel · Investigated 2026-06-30

Summary

Verdict Confirmed stale-template snapshot bug — now fixed.
Status RESOLVED Template team metadata is synced into planned orders on plan/view. See Resolution.
User-visible symptom Song Leaders appears as a scheduling target, but its schedule panel has zero cards.
Root cause Existing orders keep the old template JSON; the planner target list reads the updated live template.
Fix shape Backfill or synchronize template team metadata into already-planned orders.

Findings

CONFIRMED

Existing planned orders do not inherit teams added to their template later

Planning a month copies the current template into orders_of_service.order_json. When the template is edited later, saveTemplate updates only order_service_templates.template_json. Already-created orders remain snapshots of the older template, so they do not contain newly-required teams such as Song Leaders.

Evidence: planMonthInternal normalizes template.template and inserts it as order_json. It skips dates that already have orders. saveTemplate updates the template row only and does not touch orders_of_service rows that reference the template.
Why Counters and Ushers work
  1. The month was planned while Counters and Ushers already existed in the template.
  2. The created orders therefore contain those teams in their persisted order_json.
  3. getMonthPlan builds scheduler cards from those existing order JSON snapshots.
  4. Because Counters and Ushers are present in the snapshots, their schedule cards are available.
Why Song Leaders fails
  1. Song Leaders was added to the template after the month's orders already existed.
  2. The existing orders still lack Song Leaders in requiredTeamIds / optionalTeamIds.
  3. The Month Planner target bar includes Song Leaders because it also reads the current assigned template.
  4. The Song Leaders panel receives scheduleCards[teamId] ?? [], and there are no order-sourced cards for that team.
  5. The panel correctly renders the empty-state message from its perspective: no existing planned order cards staff Song Leaders.
ROOT CAUSE

The scheduler mixes live template targets with persisted order cards

getMonthPlan uses two different data sources for the scheduler. The actual schedule cards come from existing orders only, while the list of teams that can be opened also includes teams from the current template configuration.

Evidence: scheduleCards is populated by iterating orders and each order's order.service_type. Later, targetTeamIds is augmented by iterating settings.prepopulateDays and each current template.template.service_type.
Relevant simplified flow 
// Cards users can assign are sourced from planned orders only.
for (const order of orders) {
  for (const card of order.order.service_type) {
    const teamIds = new Set([
      ...(card.requiredTeamIds ?? []),
      ...(card.optionalTeamIds ?? []),
    ]);
    // push scheduleCards[teamId]
  }
}

// Buttons users can open are also sourced from current templates.
for (const day of settings.prepopulateDays) {
  const template = templatesById.get(day.templateId);
  for (const card of template.template.service_type) {
    for (const teamId of [...requiredTeamIds, ...optionalTeamIds]) {
      targetTeamIds.add(teamId);
    }
  }
}
IMPORTANT

A UI-only fallback would not fully fix the bug

It may be tempting to create virtual Song Leaders cards from the current template when scheduleCards is empty. That would make the scheduler list appear, but it would not make the existing order treat Song Leaders as a required team.

Evidence: saveMonthSchedule updates teamAssignments for an existing order card. It does not add the team to requiredTeamIds, optionalTeamIds, or requiredTeamCounts.
Impact
  • The order editor would still not truly know Song Leaders is required.
  • Publish gating could miss the newly-required team because the order snapshot lacks the requirement.
  • Future saves would preserve an ad-hoc assignment, not the template's required-team metadata.

Recommended fix

Synchronize template team metadata into planned orders

Add a real synchronization/backfill path for orders that were already planned from the affected template. The key is to merge team metadata, not replace entire order cards.

  1. Match existing planned order cards to current template cards by stable card.id.
  2. Add missing requiredTeamIds and optionalTeamIds from the template card into the order card.
  3. Copy missing requiredTeamCounts for newly-required teams.
  4. Preserve existing assignments, title/status, date, activities, and any order-specific edits.
  5. Prefer limiting automatic changes to orders still in Planning status, or provide an explicit “Sync template changes into planned services” action.
Where to consider wiring it
  • planMonth: after creating missing dates, also sync existing planned dates for that month.
  • getMonthPlan: optionally detect and repair stale planned orders before returning planner data.
  • saveTemplate: optionally enqueue or perform sync for future/planning orders that reference the edited template.

The least surprising user experience may be an explicit Month Planner action because template changes can be intentional for future services only. But the scheduler should not expose a team button whose panel can never save a real requirement into the existing orders.

Resolution (implemented)

RESOLVED

Planned orders are synced with their template's teams

A pure syncOrderTeamsFromTemplate(order, template) helper (src/lib/teams-logic.ts) merges template team metadata into an existing order by matching cards on their stable card.id. The merge is additive and non-destructive:

  • Unions requiredTeamIds and optionalTeamIds from the template into the order (an optional team the template now requires is promoted to required).
  • Fills in requiredTeamCounts for newly-required teams, but the order's own counts win where it already tracks one.
  • Preserves assignments, activities, title/status/date, and ad-hoc teams the planner added.
  • Teams the template later drops are left in place rather than silently removed.
  • Returns a changed flag so steady-state views perform no writes.

Wired into planMonthInternal: for each configured date with an existing order in Planning status, the order is synced and re-persisted when it changed. Published orders are never touched. getMonthPlan now runs this on every planner view (createMissing gates auto-creation to the current month only), so opening the affected month repairs the snapshot before scheduleCards are built — the Song Leaders panel populates and a saved assignment lands on an order that genuinely lists the team as required.

Files changed
  • src/lib/teams-logic.ts — new syncOrderTeamsFromTemplate helper.
  • src/lib/order-service-data.ts — sync existing Planning orders in planMonthInternal; add createMissing option; always sync in getMonthPlan.
  • src/lib/teams-logic.test.ts — unit coverage for the merge (additive, count precedence, optional→required, ad-hoc preservation, drop safety, no-op).

Suggested acceptance check

  1. Plan a month using a template that already includes Counters and Ushers.
  2. Add Song Leaders as a required team on a card in that template.
  3. Return to the already-planned month.
  4. Run the new sync/backfill path if it is explicit.
  5. Open the Song Leaders scheduler.
  6. Verify the expected service dates/cards appear instead of the empty-state message.
  7. Assign a Song Leader and save.
  8. Verify the corresponding order's order_json includes Song Leaders in requiredTeamIds with the appropriate requiredTeamCounts, not only in teamAssignments.
