Findings

Resolved Medium Month planning can throw duplicate-date errors under concurrent requests

Status: Fixed. The insert now uses ON CONFLICT(service_date) DO NOTHING with RETURNING id, and createdIds is built from the rows actually returned by the batch rather than the pre-snapshot intent.

Location: src/lib/order-service-data.ts:1380-1428

planMonthInternal first reads all existing service dates, then prepares plain INSERT statements for dates not found in that snapshot. Because service_date is unique, two simultaneous current-month planner loads or two quick Plan month submissions can both observe a missing date and then race to insert it. One request succeeds; the other can fail with a unique constraint error instead of idempotently skipping the row that appeared meanwhile.

Suggested fix: make the insert idempotent with INSERT OR IGNORE / ON CONFLICT(service_date) DO NOTHING, then compute created IDs from actual changes rather than pre-counting every intended insert.

Recommended fix:

Make each insert a no-op on conflict and use D1's RETURNING to learn which rows were actually created, so the returned createdIds reflect real inserts rather than the pre-snapshot intent (src/lib/order-service-data.ts:1408-1426): 

const statements: D1PreparedStatement[] = [];

for (const { date, dayConfig } of configuredDates) {
  if (existingDates.has(date)) {
    continue;
  }

  const template = templatesById.get(dayConfig.templateId);
  if (!template) {
    continue;
  }

  const order = normalizeTemplate(template.template, template.name);
  const title =
    dayConfig.defaultTitle.trim() || `${template.name} Order of Service`;

  statements.push(
    db
      .prepare(
        `INSERT INTO orders_of_service
           (id, title, service_type_id, service_date, status, template_id, order_json, updated_at)
         VALUES (?, ?, ?, ?, 'Planning', ?, ?, ?)
         ON CONFLICT(service_date) DO NOTHING
         RETURNING id`
      )
      .bind(
        uuidv4(),
        title,
        template.serviceTypeId,
        date,
        template.id,
        JSON.stringify(order),
        timestamp
      )
  );
}

const createdIds: string[] = [];
if (statements.length > 0) {
  const batchResults = await db.batch<{ id: string }>(statements);
  for (const result of batchResults) {
    for (const row of result.results) {
      createdIds.push(row.id);
    }
  }
}

return { createdIds };

A row lost to a concurrent insert returns no RETURNING row and is silently skipped, so the loser no longer throws a unique constraint error. The conflict target is valid: migration 0003_enforce_unique_order_service_date.sql already creates a UNIQUE INDEX on orders_of_service(service_date), which SQLite/D1 accepts as an ON CONFLICT(service_date) target.

Resolved Low Invalid month query strings pass route validation and fail in the loader

Status: Fixed. isValidMonth now requires the month component to be 01–12 (/^\d{4}-(0[1-9]|1[0-2])$/u), so validateSearch coerces values like 2026-99 to undefined and the loader falls back to the current month.

Location: src/routes/planner.tsx:94 and src/routes/planner.tsx:880-884

The client validator accepts any YYYY-MM string, so a direct visit to /planner?month=2026-99 preserves the bad value. The server later rejects it in parseMonth, which turns a malformed query param into a loader error page instead of normalizing to the current month or rejecting the search value before the loader runs.

Suggested fix: share stricter month validation with the server rule (01 through 12) or coerce invalid search params to undefined in validateSearch.

Recommended fix:

Tighten the client matcher so the month component must be 01–12, matching the server's parseMonth rule (src/routes/planner.tsx:94): 

-const isValidMonth = (value: string) => /^\d{4}-\d{2}$/u.test(value);
+const isValidMonth = (value: string) =>
+  /^\d{4}-(0[1-9]|1[0-2])$/u.test(value);

validateSearch already coerces a non-matching month to undefined (src/routes/planner.tsx:880-884), so the loader falls back to the current month instead of erroring — once isValidMonth rejects 2026-99, no further change is needed there. For a single source of truth, export this matcher (or a shared monthPattern) and reuse it in parseMonth so the two validators cannot drift.
