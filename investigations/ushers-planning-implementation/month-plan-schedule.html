Implementation Summary

Status: Implemented on branch feat/month-planner (branched off PR 1, feat/teams-and-team-members). The plan below was delivered in full; this section records exactly what shipped and where to find it.

What shipped

  • Month Planner settings — a new section on the existing /settings page lets users pick which weekdays are pre-populated and the template each uses. Sunday is enabled by default and assigned the seeded Sunday template. Saving is blocked (client and server) when an enabled weekday has no template.
  • Month Planner route at /orders/month — starts on the current month, with previous/next arrows, a month picker, a Plan Month button when configured dates are missing, a table of configured service dates, and a Schedule team button bar.
  • Automatic current-month backfill — visiting the current month creates any missing configured services (e.g. all June 2026 Sundays) without overwriting existing orders.
  • Template-driven schedule targets — the team button bar and the monthly schedule dialog are derived from the required and optional teams on each assigned template's service cards. The dialog shows every matching card across the month and supports adding and modifying existing assignments, saved in a single batched update of the affected orders' JSON.
  • Template delete protection — templates referenced by existing orders or by Month Planner settings cannot be deleted; the Templates UI surfaces the blocking reason.
  • Entry points — a “Month planner” button on the Orders list and a breadcrumb label for the route.

Executive Summary

Orders of Service are currently created and planned one at a time. The new Month Planner will add a month-first planning workflow that can pre-populate all configured service days for a selected month and then schedule teams across those services in one place.

Primary goal: make schedules that are planned a month ahead, especially ushers and counters, easy to create and revise without opening each Order of Service individually.

The first delivery should focus on Sunday services, because Sunday is pre-populated by default, must be associated to the Sunday service template, and the current month must be populated with every Sunday order.

Captured Requirements

Current Implementation

  • Orders of Service are currently planned per Sunday.
  • Orders can be generated ahead of time, but generation is currently one by one.
  • Some schedules, such as ushers and counters, are planned a month ahead and need a monthly workflow.

Month Planner

  • Orders of Service need to be categorized per month.
  • For each month that is planned, all Orders of Service should be pre-populated according to the configured weekday selection in the Month Planner section on the existing Settings page.
  • Every pre-populated weekday must have an associated template. Pre-population cannot be finalized for a weekday until a template is selected for that weekday.
  • Sunday must be pre-selected and pre-populated by default with the configured Sunday service template.
  • If other weekdays become pre-populatable, each of those weekdays must also be assigned a template before monthly planning can use them.
  • Navigating to a month that has not been planned presents a Plan Month button.
  • Selecting Plan Month pre-populates services for that month.
  • Each month needs a Schedule button bar that lets users quick-add team members to Orders of Service for the month.
  • Templates are the source of truth for scheduling targets. The quick-add workflow should target every service card in the associated template that requires or allows scheduling through required or optional teams.
  • The quick-add workflow should support teams whose schedule is planned monthly, especially ushers and counters, by reading those teams from the template associated to each pre-populated service day.
  • Selecting a team from the schedule button bar must allow users to add and modify members already assigned to schedules.

Month Selection

  • Always start on the current month.
  • Provide arrows to navigate to the previous and next month.
  • Provide a calendar button with a date picker for direct month selection.

Current Month Rule

The current month must be pre-populated for all Sundays in the month. With the current development date of 2026-06-29, the Sunday services for June 2026 are:

  • 2026-06-07
  • 2026-06-14
  • 2026-06-21
  • 2026-06-28

Proposed User Experience

Orders Month View

Add a route such as /orders/month for a month-first planning page. The page starts on the current month and includes:

  • Previous month arrow button.
  • Current selected month heading, for example June 2026.
  • Next month arrow button.
  • Calendar/month picker button.
  • Plan Month button when configured service dates are missing.
  • Schedule button bar populated from teams found in the selected month's associated templates.
  • A table of service dates in the selected month.

Plan Month Flow

  1. User navigates to a month.
  2. The loader computes all configured weekday/template pairs for that month.
  3. If a configured weekday is missing a template, block planning and show a settings error.
  4. If one or more configured dates are missing, show Plan Month.
  5. User selects Plan Month.
  6. The server creates missing Orders of Service from each date's assigned template.
  7. Existing orders are left unchanged.
  8. The page refreshes and displays all services for that month.

For the current month, the app should ensure Sunday services are present automatically or prominently require one click to create them. The preferred implementation is automatic current-month Sunday pre-population on first visit to the month planner, using the configured Sunday template.

Schedule Button Bar

The button bar should be generated from the templates associated to the selected month's pre-populated service days. If the Sunday template has service cards that require or allow Ushers and Counters, those teams appear automatically.

Selecting a team opens a monthly assignment dialog or drawer with a row for every service card in the month where that team is required or optional. Users can add, remove, and modify assigned members across the entire month.

Data Model and Settings

Recommended Settings Shape

Store Month Planning settings in the existing app_settings table under a new key such as monthPlanning. The key setting is a list of pre-populatable days. Each day must point to the template used to create orders for that weekday. 

{
  "prepopulateDays": [
    {
      "weekday": 0,
      "templateId": "default-sunday-service",
      "defaultTitle": "Sunday Order of Service"
    }
  ]
}
  • prepopulateDays: configured weekday/template pairs.
  • weekday: number where Sunday is 0.
  • templateId: required template used when creating orders for that weekday.
  • defaultTitle: title used for generated services for that weekday.
  • Quick-add schedule buttons are derived from required and optional teams in the templates assigned to the configured weekdays. This replaces a separate quickAddTeamIds setting.

If a weekday is enabled but no template is selected, the save action should fail and explain that pre-population cannot be finalized without an associated template.

Do We Need New Tables?

Not for the first implementation. Team assignments already live in each order's order_json.service_type[].teamAssignments. The month planner can update those JSON assignments in batches.

A future analytics/reporting implementation could normalize team schedule assignments into tables, but that is not necessary to satisfy the current monthly planning workflow.

Template usage does need enforcement. A template is considered in-use when it is referenced by existing Orders of Service or by a Month Planner pre-population day. In-use templates must be undeletable until those references are removed or changed.

Implementation Plan

Phase 1: Month Planning Settings

  1. Add MonthPlanningSettings types in src/lib/order-service-types.ts.
  2. Add default settings in src/lib/order-service-data.ts with Sunday selected and assigned to the default Sunday service template.
  3. Add getMonthPlanningSettings and saveMonthPlanningSettings server functions.
  4. Add a Month Planner section to the existing /settings page. Do not create a separate settings route for this feature.
  5. Allow users to select pre-populate weekdays.
  6. Require a template selection for every selected pre-populate weekday.
  7. Prevent saving/finalizing pre-population settings when any selected weekday has no template.
  8. Derive quick-add teams and schedule targets from the selected templates rather than a separate quick-add team list.

Phase 2: Month Generation Server Functions

  1. Add a date helper to calculate all dates in a month matching configured weekday/template pairs.
  2. Add getMonthPlan to load orders, settings, assigned templates, schedule targets, and team members for a selected month.
  3. Add planMonth to create missing orders for configured dates using each date's assigned template.
  4. Reject month planning if a configured pre-populate day has no valid template.
  5. Use the existing uniqueness check to avoid duplicate service dates.
  6. Batch inserts for efficiency and return the created order ids.
  7. Skip dates that already have an order instead of failing the entire month.

Phase 3: Month Planner Route

  1. Create src/routes/orders/month.tsx.
  2. Default the selected month to the current month.
  3. Support previous and next month navigation with query state, for example ?month=2026-06.
  4. Add a calendar/month picker for direct month selection.
  5. Show Plan Month when configured service dates are missing.
  6. Show existing orders and missing dates in a clear table.
  7. Link each planned order to its existing order editor.

Phase 4: Quick Schedule Teams

  1. Render quick-add team buttons from the required and optional teams found in the templates assigned to the month's pre-populated days.
  2. Create a monthly team schedule dialog component.
  3. For each order in the month, show every service card where the selected team is required or optional by that order's source template.
  4. Allow users to modify assigned members using existing team member selection patterns.
  5. Persist changes by updating the affected orders' order_json.
  6. Invalidate the route after save so the month view reflects changes.

Phase 5: Current Month Sunday Backfill

  1. On first visit to the month planner, detect the current month.
  2. Ensure every Sunday in the current month has an Order of Service.
  3. Do not overwrite existing orders.
  4. Use the configured Sunday template and title.
  5. For June 2026, ensure 2026-06-07, 2026-06-14, 2026-06-21, and 2026-06-28.

Phase 6: Template Delete Protection

  1. Update deleteTemplate to check existing Orders of Service and Month Planner day settings before deletion.
  2. If a template is referenced by any order or pre-populate day, block deletion with a clear error message.
  3. In the Templates UI, show an "In use" indicator and disable or confirm-block deletion for referenced templates.
  4. Allow deletion only after the user moves those weekdays/orders to a different template or removes the pre-populate day.

Core Date Helper Pseudocode

const getMonthDatesForDayConfigs = (
  month: string,
  dayConfigs: MonthPlanningDayConfig[]
): Array<{ date: string; dayConfig: MonthPlanningDayConfig }> => {
  const [year, monthNumber] = month.split("-").map(Number);
  const date = new Date(Date.UTC(year, monthNumber - 1, 1));
  const dates: Array<{ date: string; dayConfig: MonthPlanningDayConfig }> = [];

  while (date.getUTCMonth() === monthNumber - 1) {
    const dayConfig = dayConfigs.find(
      (config) => config.weekday === date.getUTCDay()
    );

    if (dayConfig) {
      dates.push({ date: date.toISOString().slice(0, 10), dayConfig });
    }

    date.setUTCDate(date.getUTCDate() + 1);
  }

  return dates;
};

Batch Update Strategy for Team Assignments

The monthly scheduler should update only the selected team's assignments and preserve every other part of the order JSON, including hymns, activities, and other teams.

  1. Load affected orders by id.
  2. For each order, locate the configured card or cards that support the selected team.
  3. Replace that team's memberIds with the submitted member ids.
  4. Normalize using existing team helpers where possible.
  5. Save all changed orders in a D1 batch.

Decisions and Remaining Questions

Decision made: templates are the source of truth for monthly scheduling targets

The monthly quick-add scheduler should not use a hard-coded service card and should not rely on a separate quick-add team mapping. The template assigned to each pre-populated weekday determines which service cards need scheduling. If the Sunday service is pre-populated from the Sunday Service template, then that Sunday template feeds the schedule targets for Sunday orders. If another weekday becomes pre-populatable, that weekday must also have a template, and that template feeds its schedule targets.

Resolved targeting rule

  1. Settings define which weekdays can be pre-populated.
  2. Every pre-populated weekday must be assigned a template.
  3. The template creates the Order of Service for that date.
  4. Required and optional teams on that template's service cards become monthly schedule targets.
  5. Quick-add can assign members to all matching service cards, not just one card.

Template deletion rule

A template becomes protected when it is used by existing Orders of Service or by Month Planner pre-population settings. The user must first move those references to another template or remove the weekday from pre-population before the template can be deleted.

Should planning happen automatically or by button?

Future months should use the explicit Plan Month button. The current month has a special requirement to be pre-populated for all Sundays; automatic current-month Sunday backfill is recommended.

Where should Month Planner settings live?

Decision made: Month Planner settings should be a section on the existing /settings page, not a separate route. The section should sit alongside the email configuration and recipient settings sections.
