BLOCKER ✅ Resolved

Required-team publishing can be bypassed by stale or invalid member IDs

The server-side publish gate only checks whether a required team assignment has at least one raw ID in order_json. It does not verify that the member exists or still belongs to that team. Membership changes and member deletes update relational tables, but do not clean orders_of_service.order_json.

Evidence: findMissingRequiredTeams checks only getAssignmentMemberIds(card, teamId).length === 0 at src/lib/teams-logic.ts:253-262. assertRequiredTeamsStaffed loads teams only, not members or memberships, at src/lib/order-service-data.ts:736-742. saveTeamMember and deleteTeamMember mutate team_member_teams/team_members at src/lib/order-service-data.ts:2154-2184 and 2195-2199 without touching existing orders.
Impact and fix
  • Repro: assign Alice to required Ushers on an order, then remove Alice from Ushers or delete Alice. The stale Alice ID remains in teamAssignments; server publish still passes.
  • UI mismatch: the row resolves assigned avatars from all members, while the dialog only lists current team members (src/components/order-team-assignment.tsx:314-319, 420-422), so stale assignments can be invisible or impossible to uncheck.
  • Smallest safe fix: on publish, validate every assigned memberId with a DB join against team_member_teams for the same teamId. Also prune invalid IDs in saveOrder and after member/team membership mutations.
Proposed fix

Make the publish gate count only members who currently belong to the required team, and keep order_json tidy so the UI and the gate agree.

  1. Load live memberships server-side. Add loadTeamMemberIds(db) beside loadTeamsById in src/lib/order-service-data.ts, returning a Map<teamId, Set<memberId>> from a single SELECT team_id, member_id FROM team_member_teams.
  2. Validate membership in the gate. Give findMissingRequiredTeams (src/lib/teams-logic.ts:253) an optional membersByTeam param and count only valid members: 
    const assigned = getAssignmentMemberIds(card, teamId);
                                    const validCount = membersByTeam
                                    ? assigned.filter((id) => membersByTeam.get(teamId)?.has(id)).length
                                    : assigned.length;
                                    if (validCount < REQUIRED_TEAM_MINIMUM) { /* missing */ }
    The param is optional so the existing unit tests stay green; the server path always supplies it.
  3. Wire it into assertRequiredTeamsStaffed (order-service-data.ts:736): Promise.all([loadTeamsById(db), loadTeamMemberIds(db)]) then pass membersByTeam to findMissingRequiredTeams.
  4. Prune stale IDs on save. In saveOrder (order-service-data.ts:1298), after normalizeTemplate, drop assigned memberIds not present in membersByTeam.get(teamId) before persisting. This heals existing orders the next time they are saved and removes the row/dialog mismatch at its source.
  5. Fix the dialog mismatch. In order-team-assignment.tsx, build the dialog member list from the union of membersForTeam(teamId) and any currently-assigned member, so a stale assignment is always visible and can be unchecked even after pruning lands.
  6. Test. Add a server-side test asserting publish is blocked when a required team's only assigned member has been removed from that team, plus a teams-logic unit test for findMissingRequiredTeams with a stale id.
HIGH ✅ Resolved

Deleting a team leaves orphaned template/order JSON references

Team IDs are stored inside template and order JSON arrays, so D1 foreign keys cannot protect those references. The delete path removes the team row and memberships only. Required teams can become impossible to staff or remove through the UI; optional teams silently become dead metadata.

Evidence: deleteTeam only nulls child parent pointers and deletes from team_member_teams/teams at src/lib/order-service-data.ts:2066-2073. The app already knows templates can reference teams in JSON because getTeamTemplates scans requiredTeamIds and optionalTeamIds at src/lib/order-service-data.ts:2079-2103, but deletion does not consult that scan.
Impact and fix
  • Repro: mark Ushers required in a template, then delete Ushers. The template editor no longer lists that team, so the stale required ID cannot be removed normally. New orders inherit the orphaned requirement.
  • Cloudflare/D1 note: D1 foreign keys enforce relational columns, but these JSON fields are outside that protection.
  • Smallest safe fix: block deleting any team referenced by templates or planning/published orders, or do a transactional JSON cleanup across order_service_templates.template_json and orders_of_service.order_json. Validate saved templates/orders against existing teams.
Proposed fix

Block the delete when the team is still referenced from JSON, since D1 foreign keys cannot. Refusing is the smallest no-data-loss option; the planner removes the references first, then deletes.

  1. Add a reference scan findTeamReferences(db, teamId) in src/lib/order-service-data.ts that loads template_json from order_service_templates and order_json from orders_of_service, parses each, and collects names where any card references the team. Reuse the same predicate as getTeamTemplates but extend it to teamAssignments too: 
    const cardUsesTeam = (card) =>
                                    (card.requiredTeamIds ?? []).includes(teamId) ||
                                    (card.optionalTeamIds ?? []).includes(teamId) ||
                                    (card.teamAssignments ?? []).some((a) => a.teamId === teamId);
  2. Guard deleteTeam (order-service-data.ts:2060): call the scan before the db.batch; if anything references the team, throw a clear error listing the offending templates/orders ("Remove this team from N template(s) and M order(s) before deleting it: …").
  3. Surface the error in the UI. The team editor's delete handler already shows server errors via toast, so no new UI is needed — the planner sees exactly where the team is used.
  4. Alternative (follow-up, not this PR): a transactional JSON cleanup that strips the id from requiredTeamIds/optionalTeamIds and drops matching teamAssignments across all templates and orders. Deferred because silently editing published orders is riskier than refusing the delete.
  5. Test. Server test: deleting a referenced team throws and leaves the row intact; deleting an unreferenced team succeeds.
HIGH ✅ Resolved

Optional teams configured on templates are not honored in the order assignment UI

The template editor lets a team be marked optional, but the order editor neither displays optional teams nor restricts the add menu to them. Optional-team configuration becomes mostly invisible; the user sees a generic empty table and an all-teams menu.

Evidence: getCardTeamIds includes requiredTeamIds and existing teamAssignments, but omits optionalTeamIds at src/lib/teams-logic.ts:155-163. OrderTeamAssignment uses optionalTeamIds only to decide whether the section exists (src/components/order-team-assignment.tsx:300-303), then builds availableTeams from every team not already shown (311-313).
Impact and fix
  • Repro: set Singers to Optional for a service card, create an order, and open Team assignments. Singers is not shown; Add team lists all teams with no optional marker.
  • Smallest safe fix: include optional teams in getCardTeamIds as non-required rows, or render a separate optional-suggestions list. If arbitrary extra teams are allowed, label that as an explicit “Other teams” path instead of conflating it with template optional teams.
Proposed fix

Render template optional teams as real (non-required) rows and label them, so the configuration is visible and the “Add team” menu stops re-offering them.

  1. Include optional teams in getCardTeamIds (src/lib/teams-logic.ts:159): 
    export const getCardTeamIds = (card) =>
                                    uniqueStrings([
                                    ...(card.requiredTeamIds ?? []),
                                    ...(card.optionalTeamIds ?? []),
                                    ...(card.teamAssignments ?? []).map((a) => a.teamId),
                                    ]);
    Optional teams now appear as rows and are excluded from availableTeams automatically, fixing both halves of the report.
  2. Add isTeamOptional(card, teamId) next to isTeamRequired and show an Optional badge on those rows in order-team-assignment.tsx.
  3. Gate the Remove button. Disable Remove for both required and optional (template-configured) teams — only ad-hoc teams a planner added are removable. Otherwise removing an optional team just clears its assignment and the row reappears on re-render (it is still in optionalTeamIds).
  4. Clarify the menu. Label the “Add team” dropdown header “Other teams” to make clear it adds teams beyond the template's required/optional set, rather than conflating ad-hoc additions with optional config.
  5. Test. teams-logic unit test: getCardTeamIds returns optional ids; component/smoke test that an optional team renders with the Optional badge and is absent from the add menu.
MEDIUM ✅ Resolved

Team hierarchy invariants are client-only

The UI tries to keep the hierarchy shallow by offering only top-level teams as parents, but the server accepts any parent ID except self. A direct server-function call, stale tab, or future UI can create child-of-child structures or cycles that the tree builder does not handle.

Evidence: saveTeam only rejects parentTeamId === id at src/lib/order-service-data.ts:2038-2043. The client-side parent picker filters to top-level options at src/components/team-editor-page.tsx:94-99, but that is not an integrity boundary. buildTeamTree assumes roots and direct children at src/lib/teams-logic.ts:67-90.
Impact and fix
  • Repro: call saveTeam for A with parent B and B with parent A. Both pass the application-level validation; the member editor tree can lose those teams because there are no roots.
  • Smallest safe fix: in saveTeam, verify the parent exists, has no parent, and that the update cannot create a cycle. Add a unit test for cycle/depth rejection.
Proposed fix

Enforce the two-level hierarchy on the server. Because the tree is only ever two levels deep, two structural rules make cycles impossible without graph traversal: a parent must itself be a root, and a team that already has children cannot be re-parented.

  1. Extract a pure validator validateTeamParent({ id, parentTeamId, teams }) in src/lib/teams-logic.ts returning an error string or null, so it is unit-testable without D1. Rules: parent exists; parent is not id; parent has no parentTeamId; id has no existing children.
  2. Call it in saveTeam (src/lib/order-service-data.ts:2027) after computing parentTeamId, fetching the candidate set in one query: 
    if (parentTeamId) {
                                    const parent = await db
                                    .prepare("SELECT id, parent_team_id FROM teams WHERE id = ?")
                                    .bind(parentTeamId).first();
                                    if (!parent) throw new Error("Parent team does not exist.");
                                    if (asString(parent.parent_team_id))
                                    throw new Error("Teams can only nest one level deep.");
                                    const hasChildren = await db
                                    .prepare("SELECT 1 FROM teams WHERE parent_team_id = ? LIMIT 1")
                                    .bind(id).first();
                                    if (hasChildren)
                                    throw new Error("A team with sub-teams cannot become a sub-team.");
                                    }
    The existing parentTeamId === id check stays.
  3. Keep the client filter in team-editor-page.tsx as a convenience, now backed by a real server boundary.
  4. Test. Unit-test validateTeamParent for the cycle case (A→B, B→A), the depth case (parent already has a parent), and the has-children case; assert each is rejected.
